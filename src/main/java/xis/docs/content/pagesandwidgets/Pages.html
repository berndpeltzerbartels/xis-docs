Pages are the entry points of your XIS application. They are controllers annotated with @Page that map to specific URLs and render complete HTML pages in response to HTTP requests.

Defining a Page

A page controller is a Java class annotated with @Page("/path"), where the path parameter specifies the URL at which the page will be accessible. Each page controller has a corresponding HTML template file located in the same package following XIS's convention-based approach. 

@Page("/hello")
public class HelloPage {
    
    @ModelData
    public String greeting() {
        return "Hello World!";
    }
}

The corresponding HelloPage.html template would be placed in the same package and can access the model data provided by the controller: 

<xis:template xmlns:xis="http://www.w3.org/1999/XSL/Transform">
    <h1>${greeting}</h1>
</xis:template>
Page Lifecycle

When a user navigates to a page's URL, XIS processes the request through the following steps:

  • The ControllerService routes the request to the appropriate page controller
  • Controller methods annotated with @ModelData are invoked to provide data
  • The HTML template is rendered with the provided model data
  • The resulting HTML is sent as the HTTP response
Page vs Widget

Unlike widgets, pages are directly accessible via URL and represent complete, standalone views. Pages can embed widgets to compose complex interfaces from reusable components.

Important: JavaScript Inclusion
Pages and Widgets in XIS cannot include JavaScript files using standard HTML <script> tags directly in the template. This is because XIS is a Single Page Application (SPA) where content is dynamically inserted into the DOM at runtime. Browsers do not execute <script> tags that are added to the DOM after the initial page load - this is an important browser security feature, but it requires a different approach for SPAs.

Instead, XIS uses a convention-based approach where you place a .js file with the same name as your controller in the same package. These files are automatically discovered and included. You can also use the @JavaScriptFile annotation for explicit control. Other resources like CSS can be included directly in the template using standard HTML tags. See the Custom Assets documentation for details.
Deep Linking Support

XIS pages are inherently deep-link capable - each page is accessible via its unique URL path specified in the @Page annotation. Additionally, widgets embedded in pages can also participate in deep linking by specifying their own url, title, and containerId in the @Widget annotation.

This enables powerful navigation scenarios where users can bookmark specific widget states within a page, and the application will correctly restore both the page and the widget to the expected state when the URL is accessed. 

// Page provides the foundation
@Page("/dashboard")
class DashboardPage { }

// Widget adds deep-linkable state
@Widget(id = "Statistics", 
       url = "/dashboard/stats/{period}",
       title = "Statistics")
class StatisticsWidget { }

For detailed information on widget deep linking, including containerId for container control, see the Widget Details section.
