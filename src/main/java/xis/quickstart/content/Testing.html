XIS makes it easy to write fast integration tests for both Spring Boot and XIS Boot applications. The dependency injection is performed even for Spring components, but it's important to know that this is not a real Spring context. Instead, XIS creates a lightweight testing context that supports Spring annotations but doesn't provide all Spring features.

When creating the context, you can pass both classes (which will be instantiated) and objects (such as mocks). This flexibility makes testing with mocked dependencies very straightforward.

Testing Hello World

Let's write a test for our Hello World page. Create a test file src/test/java/quickstart/helloworld/HelloWorldTest.java: 

package quickstart.helloworld;

import one.xis.context.IntegrationTestContext;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;

import static org.assertj.core.api.Assertions.assertThat;

class HelloWorldTest {

    private IntegrationTestContext context;

    @BeforeEach
    void init() {
        context = IntegrationTestContext.builder()
                .withSingleton(HelloWorld.class)
                .build();
    }

    @Test
    void showGreeting() {
        var result = context.openPage("/hello-world.html");

        var document = result.getDocument();
        assertThat(document.getElementByTagName("title")).isNotNull();
        assertThat(document.getElementByTagName("h1")).isNotNull();

        var titleElement = document.getElementByTagName("title");
        var h1Element = document.getElementByTagName("h1");

        assertThat(titleElement.getInnerText()).isEqualTo("Hello, World!");
        assertThat(h1Element.getInnerText()).isEqualTo("Hello, World!");

    }
}

The result object contains the Document and various framework-specific utilities. The DOM API provides many methods for verifying the result and triggering events (like clicks).

⚠️ Important
openPage can only be called once per context. If you need to navigate further, you must use element interactions from the document, for example clicking on elements to navigate to other pages or submit forms. However, it is recommended not to put too much into a single test, but rather to write multiple focused tests.
