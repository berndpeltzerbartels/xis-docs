Architecture / Microfrontends

Microfrontend Architecture

XIS fits vertical ownership: a service can own its business logic and the page or frontlet that presents it. The browser can still behave like one coherent application.

Vertical slices

XIS microfrontends are not primarily a packaging trick. They are a way to keep business ownership vertical: persistence, service code, page controller, template, validation, and tests can live in the same feature or service boundary. The browser receives a coherent UI, while the code remains split along business responsibilities.

Think of a product page in a shop. The visible page is one experience, but catalog, price, discount, stock, store-pickup, notification, and cart behavior may belong to different teams and databases. XIS frontlets let those teams contribute focused UI parts without forcing the product page to become a large client-side integration project.

The product id belongs to the page URL, not to the frontlet instances. The page and every frontlet can read the same path variable with @PathVariable("id"). Frontlet parameters are still useful, but they should describe the concrete frontlet instance, not repeat context that is already part of the route.

<article class="product-page">
  <header class="product-hero">

    <h1>${product.name}</h1>
    <p>${product.teaser}</p>
  </header>

  <main xis:frontlet-container="description"
        xis:default-frontlet="ProductDescriptionFrontlet"></main>

  <aside xis:frontlet-container="offer"
         xis:default-frontlet="DeliveryAndDiscountFrontlet"></aside>

  <section xis:frontlet-container="stock"
           xis:default-frontlet="StockAndPickupFrontlet"></section>

  <section xis:frontlet-container="cart"
           xis:default-frontlet="AddToCartFrontlet"></section>
</article>

ProductPage.java

import one.xis.ModelData;
import one.xis.Page;
import one.xis.PathVariable;

@Page("/products/{id}.html")
class ProductPage {

    private final CatalogService catalogService;

    ProductPage(CatalogService catalogService) {
        this.catalogService = catalogService;
    }

    @ModelData("product")
    ProductHeader product(@PathVariable("id") long productId) {
        return catalogService.header(productId);
    }
}

ProductDescriptionFrontlet.java

import one.xis.Frontlet;
import one.xis.ModelData;
import one.xis.PathVariable;

@Frontlet
class ProductDescriptionFrontlet {

    private final CatalogService catalogService;

    ProductDescriptionFrontlet(CatalogService catalogService) {
        this.catalogService = catalogService;
    }

    @ModelData("description")
    ProductDescription description(@PathVariable("id") long productId) {
        return catalogService.description(productId);
    }
}

StockAndPickupFrontlet.java

import one.xis.Action;
import one.xis.FormData;
import one.xis.Frontlet;
import one.xis.ModelData;
import one.xis.PathVariable;

@Frontlet
class StockAndPickupFrontlet {

    private final InventoryService inventoryService;

    StockAndPickupFrontlet(InventoryService inventoryService) {
        this.inventoryService = inventoryService;
    }

    @ModelData("stock")
    StockStatus stock(@PathVariable("id") long productId) {
        return inventoryService.status(productId);
    }

    @FormData("notification")
    NotificationForm notification() {
        return new NotificationForm();
    }

    @Action
    void notifyWhenAvailable(@PathVariable("id") long productId,
                             @FormData("notification") NotificationForm form) {
        inventoryService.registerNotification(productId, form.email());
    }
}

AddToCartFrontlet.java

import one.xis.Action;
import one.xis.FormData;
import one.xis.Frontlet;
import one.xis.PathVariable;

@Frontlet
class AddToCartFrontlet {

    private final CartService cartService;

    AddToCartFrontlet(CartService cartService) {
        this.cartService = cartService;
    }

    @FormData("cartLine")
    CartLineForm cartLine() {
        return new CartLineForm(1);
    }

    @Action
    void addToCart(@PathVariable("id") long productId,
                   @FormData("cartLine") CartLineForm form) {
        cartService.add(productId, form.quantity());
    }
}

The catalog, pricing, inventory, and checkout teams can each own one vertical slice: their persistence, services, controller, template, validation, and tests. The product page stays readable because the route supplies the common product context, while each frontlet decides what it needs to render and refresh.

Pages and frontlets across modules

A page can be owned by one module and composed with frontlets from another. This is useful for dashboards, shell applications, and distributed business tools. The important rule is that each fragment should still have a clear owner and a clear refresh contract. 

xis.distributed.hosts=https://catalog.example.com,https://inventory.example.com,https://checkout.example.com
xis.distributed.allowed-origins=https://shop.example.com

Routing model

Distributed applications need predictable routing. URLs should stay meaningful and should not depend on fragile client-side state. XIS can navigate by URL, by page response, and by frontlet refresh, so the architecture can use browser-visible routes where they help and server-side interactions where they keep the workflow simpler.

SSO

Authentication is the place where distributed applications become visible to users. XIS security and SSO support belong together with the microfrontend story, but the detailed security model lives in the security chapter. The architectural point is simple: separate services should not force separate logins for one logical application.

When not to split

A small application does not become better just because every screen is a separate artifact. Split when the team structure, deployment model, or business boundary needs it. Otherwise, a single XIS application with well-kept feature packages is often easier to understand.
