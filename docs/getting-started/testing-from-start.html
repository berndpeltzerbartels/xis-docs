Testing from the start

XIS should make test-driven development feel natural. A controller and template form a small slice that can be rendered, interacted with, and verified without constructing a separate frontend test universe.

The early testing promise

The first tutorial includes a test before the reader has learned the entire framework. This makes the product story clear: XIS does not hide behavior behind a browser-only client application that is hard to exercise from JVM tests.

The important point is not only that XIS has tests. The point is that the ordinary programming model is testable: open a XIS page, inspect the rendered document, click an action, submit a form, and assert the resulting DOM or service state. You do not need to build a parallel REST client and a parallel frontend state model just to test a page workflow.

Generated starter tests

If the project uses the XIS Gradle plugin, xisTests can generate starter tests for page controllers. This is useful at the beginning of a page because it creates the test file in the right package, wires the XIS test context, and opens the page URL from the page annotation. 

./gradlew xisTests

A practical workflow is to sketch the controller first, including the @ModelData, @FormData, and @Action methods that define the page contract, then generate templates and tests: 

./gradlew xisTemplates xisTests

Generated tests are starting points. They should be edited immediately until they describe the behavior you want. The generator avoids boilerplate; it does not replace test design. 

package example.dashboard;

import one.xis.boot.test.XisBootTest;
import one.xis.context.IntegrationTestContext;
import one.xis.test.InTestContext;
import org.junit.jupiter.api.Test;

import static org.junit.jupiter.api.Assertions.assertNotNull;

@XisBootTest
class DashboardPageTest {

    private IntegrationTestContext context;

    @InTestContext
    private DashboardPage dashboardPage;

    @Test
    void opensDashboard() {
        var client = context.openPage("/dashboard.html");

        assertNotNull(client.getDocument());
    }
}

Using the annotation without generated tests

The generated style is not special to generated files. You can write the same kind of test by hand. Use @XisBootTest when you want the JUnit extension to create the XIS integration-test context and support XIS test annotations such as @InTestContext, @Mock, @Spy, and @Captor. 

package example.products;

import one.xis.boot.test.XisBootTest;
import one.xis.context.IntegrationTestContext;
import one.xis.test.InTestContext;
import org.junit.jupiter.api.Test;

import static org.junit.jupiter.api.Assertions.assertEquals;

@XisBootTest
class ProductPageTest {

    private IntegrationTestContext context;

    @InTestContext
    private ProductService productService;

    @Test
    void opensProductPage() {
        productService.add(new Product(42, "Desk"));

        var client = context.openPage("/products/42.html");
        var document = client.getDocument();

        assertEquals("Desk", document.getElementByTagName("h1").getInnerText());
    }
}

@InTestContext registers a real collaborator in the XIS context. The controller sees the same instance through constructor or field injection that the test later inspects.

Testing interactions

Tests should not stop at "the page opens." XIS tests can interact with the document. A button click can call a server action, a form submit can trigger validation, and the returned DOM can be asserted. 

@XisBootTest
class CounterPageTest {

    private IntegrationTestContext context;

    @Test
    void incrementsCounter() {
        var client = context.openPage("/counter.html");
        var document = client.getDocument();

        assertEquals("0", document.getElementById("count").getInnerText());

        document.getElementById("increment").click();

        assertEquals("1", client.getDocument().getElementById("count").getInnerText());
    }
}

The exact document helper methods depend on the element you are testing. The mental model is stable: inspect the rendered document, perform the interaction a user would perform, then inspect the updated document or an injected collaborator.

Manual integration context

You can also write XIS integration tests without @XisBootTest. This is useful for very explicit tests, for builds that do not use the starter dependency, or for cases where the test wants complete control over the registered classes and instances. 

package example.products;

import one.xis.context.IntegrationTestContext;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;

import static org.junit.jupiter.api.Assertions.assertEquals;

class ProductPageManualTest {

    private IntegrationTestContext context;
    private ProductService productService;

    @BeforeEach
    void setUp() {
        productService = new ProductService();
        context = IntegrationTestContext.builder()
                .withBasePackageClass(ProductPageManualTest.class)
                .withInstance(productService)
                .build();
    }

    @Test
    void opensProductPage() {
        productService.add(new Product(42, "Desk"));

        var client = context.openPage("/products/42.html");

        assertEquals("Desk", client.getDocument().getElementByTagName("h1").getInnerText());
    }
}

This lower-level style is still a XIS integration test. It renders the page through the XIS controller/template lifecycle. It simply does not use the JUnit annotation as a convenience layer.

What belongs where

Test kindUse it for
Unit testServices, repository-adjacent logic, formatters, validators.
XIS integration testPages, actions, validation, navigation, modal flows, drag and drop.
Browser/E2E testReal browser behavior, CSS/JS integration, deployment-level confidence.

Generated is not required

There are three valid testing styles:

  • Generate a starter with xisTests, then edit it.
  • Write a hand-made @XisBootTest.
  • Build an IntegrationTestContext manually without the annotation.

The first style is fastest when starting a page. The second is concise for ordinary page and frontlet behavior. The third is best when the test setup itself is part of what you want to control.

Documentation rule

Every non-trivial code example should either live in an executable documentation project or be derived from code that is compiled and tested somewhere. The same discipline is useful in application documentation: executable examples age much better than copied snippets.

The detailed reference for generated tests, document interaction, drag and drop, authentication, manual contexts, and test dependencies lives in the Examples and Tests guide.
