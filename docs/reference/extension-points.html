Reference / Extension points

Extension points

Advanced extension points include aspects, interface advice, custom proxies, request filters, HTTP filters, custom JavaScript, and custom EL functions.

Use extension points sparingly

Extension points exist so applications and libraries can integrate with the framework without forking it. They are not meant to be the first solution for normal page behavior. Prefer ordinary pages, actions, repositories, and services until the repeated pattern is clear.

Filters

HTTP filters are the right place for cross-cutting request handling such as static resource shortcuts, diagnostic headers, or special integration endpoints. Page logic should stay in page controllers. 

import one.xis.context.Component;
import one.xis.http.FilterChain;
import one.xis.http.HttpFilter;
import one.xis.http.HttpRequest;
import one.xis.http.HttpResponse;

import java.util.UUID;

@Component
class RequestIdFilter implements HttpFilter {
    @Override
    public void doFilter(HttpRequest request, HttpResponse response, FilterChain chain) {
        response.addHeader("X-Request-Id", UUID.randomUUID().toString());
        chain.doFilter(request, response);
    }
}

Custom EL functions

Custom expression-language functions can make templates cleaner when a formatting or calculation rule is repeated. They should remain deterministic and presentation-oriented. Business decisions belong in model data or controller methods. 

# src/main/resources/META-INF/xis/js/extensions
js/template-functions.js
XIS.addElFunction('initials', function(name) {
    return String(name || '')
        .trim()
        .split(/\s+/)
        .map(part => part.charAt(0).toUpperCase())
        .join('');
});
<span class="avatar">${initials(customer.name)}</span>

Proxies and aspects

Proxy and advice mechanisms are advanced integration tools. They are useful for infrastructure concerns, but they can also make control flow harder to read. Reach for them after the ordinary component model is clear, not as the default way to structure application code. 

import one.xis.context.Advice;
import one.xis.context.AdviceInvocation;
import one.xis.context.UseAdvice;

import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;

@UseAdvice(TimedAdvice.class)
@Retention(RetentionPolicy.RUNTIME)
@interface Timed {
}

class TimedAdvice implements Advice {
    @Override
    public Object around(AdviceInvocation invocation) throws Throwable {
        long start = System.nanoTime();
        try {
            return invocation.proceed();
        } finally {
            System.out.println(invocation.method().getName() + " took " + (System.nanoTime() - start) + " ns");
        }
    }
}

Replaceable defaults

Framework modules and reusable libraries can provide a fallback implementation with @DefaultComponent. If the application provides a normal component for the same type, the normal component wins. This is useful when a module should work out of the box but still be easy to replace. 

import one.xis.context.DefaultComponent;

@DefaultComponent
class DefaultMailSender implements MailSender {
    @Override
    public void send(Message message) {
        // framework or library default
    }
}
import one.xis.context.Component;

@Component
class SmtpMailSender implements MailSender {
    @Override
    public void send(Message message) {
        // application-specific implementation
    }
}

The replacement rule also applies to injected collections and to @Bean methods declared on a replaced default component. That keeps defaults from leaking into the runtime after the application has supplied a real implementation.
