Application security

The ordinary security story is application-level security: pages and actions can require roles, templates can hide or show content based on authorization, DTOs can carry role metadata, and ownership checks can decide whether a user may access a particular object.

Roles come from the authenticated user context. In a local-login application they are supplied by the local credential/user service. With an external OpenID Connect provider they are mapped from the authenticated identity and its claims. Once the user is authenticated, page and action checks read those roles through the XIS security context. 

import one.xis.Action;
import one.xis.ModelData;
import one.xis.Page;
import one.xis.Roles;
import one.xis.UserId;

@Page("/account.html")
@Roles("USER")
class AccountPage {

    @ModelData("message")
    String message(@UserId String userId) {
        return "Account for " + userId;
    }

    @Action
    @Roles("ADMIN")
    void rebuildIndex() {
        adminService.rebuildIndex();
    }
}
<button xis:if="isUserInRole('ADMIN')" xis:action="rebuildIndex">
  Rebuild index
</button>

Template visibility is only a usability rule. The action still needs @Roles or @Authenticated, because browser HTML is never the security boundary.
