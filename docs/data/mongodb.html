MongoDB

MongoDB support gets its own chapter: dependencies, configuration, document mapping, repositories, JSON queries, and change streams.

Documents

MongoDB works best when the aggregate shape already fits the document model. XIS documents mapping, identifiers, repository operations, and the difference between document updates and relational joins without pretending that MongoDB is just SQL with another syntax. 

import one.xis.mongodb.MongoDocument;
import one.xis.mongodb.MongoField;
import one.xis.mongodb.MongoId;

@MongoDocument("customers")
class Customer {
    @MongoId
    String customerId;

    @MongoField("first_name")
    String firstName;

    String lastName;
}

Repositories

Repository methods should make persistence explicit. Loading a document and changing a Java object should not silently write data back. This keeps the mental model aligned with services, DTOs, and server-driven UI actions. 

import one.xis.mongodb.MongoCrudRepository;
import one.xis.mongodb.MongoQuery;
import one.xis.mongodb.MongoRepository;

import java.util.List;

@MongoRepository
interface CustomerRepository extends MongoCrudRepository<Customer, String> {

    @MongoQuery("{ lastName: ?0 }")
    List<Customer> findByLastName(String lastName);
}

Change streams

MongoDB change streams can pair well with XIS refresh events. A change in the database can invalidate a page or frontlet, and the browser can then reload the affected model through the normal XIS lifecycle. 

import one.xis.RefreshEvent;
import one.xis.RefreshEventPublisher;
import one.xis.context.Component;
import one.xis.mongodb.MongoChangeEvent;
import one.xis.mongodb.MongoWatch;

@Component
class CustomerChanges {
    private final RefreshEventPublisher refreshEvents;

    CustomerChanges(RefreshEventPublisher refreshEvents) {
        this.refreshEvents = refreshEvents;
    }

    @MongoWatch(Customer.class)
    void customerChanged(MongoChangeEvent<Customer> event) {
        refreshEvents.publish(new RefreshEvent("customers"));
    }
}

Current maturity

MongoDB is part of the XIS documentation because the framework has support and tests for it. The main production path has historically been SQL, so treat MongoDB integration like any newer persistence path: keep application tests close to the repositories and change-stream behavior you rely on.
