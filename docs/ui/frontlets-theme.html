UI Composition / Frontlets and theme

Frontlets and theme

Frontlets, containers, includes, reusable artifacts, replaceable templates, and the optional XIS theme belong together: they are the composition layer before an application becomes distributed.

Frontlets

A frontlet should own a small, coherent part of the UI: navigation, a summary widget, a list fragment, a toolbar, or a reusable edit block. It has its own model data and can refresh without replacing the whole page.

Includes and templates

Includes are useful for shared markup that does not need its own controller model. Frontlets are better when the fragment has behavior, data, or refresh semantics. Keeping that distinction clear prevents small template helpers from turning into hidden application modules.

Theme

XIS should provide enough theme structure that applications start with a coherent look, but applications must be able to override styling. The theme is a default, not an ownership claim over every visual decision.

Drag and drop

Drag-and-drop interactions should still end in server actions. The browser can carry a small payload, but the server decides whether the move is allowed and what must refresh afterwards. This keeps the interaction testable in the same way as links and forms.

Example: product page as team boundary

A shop product page is a good example because everything revolves around the same product, but the data usually belongs to different systems and teams. Catalog data, current stock, delivery price, discounts, store pickup, and cart handling are not necessarily owned by the same service.

The page can still look like one product page. XIS can compose it from specialized frontlets that all read the product id from the page URL with @PathVariable("id"). The product id is therefore not repeated as a frontlet parameter. 

<h1>${product.name}</h1>

<main xis:frontlet-container="description"
      xis:default-frontlet="ProductDescriptionFrontlet"></main>

<aside xis:frontlet-container="offer"
       xis:default-frontlet="DeliveryAndDiscountFrontlet"></aside>

<section xis:frontlet-container="stock"
         xis:default-frontlet="StockAndPickupFrontlet"></section>

<section xis:frontlet-container="cart"
         xis:default-frontlet="AddToCartFrontlet"></section>
@Page("/products/{id}.html")
class ProductPage {

    @ModelData("product")
    ProductHeader product(@PathVariable("id") long productId) {
        return catalogService.header(productId);
    }
}

@Frontlet
class StockAndPickupFrontlet {

    @ModelData("stock")
    StockStatus stock(@PathVariable("id") long productId) {
        return inventoryService.status(productId);
    }
}

The catalog team can own the product description. The logistics team can own stock and pickup. The pricing team can own discounts and delivery price. The checkout team can own the add-to-cart workflow. This makes a frontlet an organizational pattern as much as a UI pattern: it is a possible team slice.

This is optional. A XIS application can also be built from ordinary pages without frontlets and still behave like an SPA. Frontlets become valuable when a screen naturally contains independently owned or independently refreshed parts.
