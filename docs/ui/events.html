UI Composition / Events and refresh

Events and refresh

XIS refresh events are invalidation signals, not a second business API. The browser receives a key and reloads the affected page or frontlet through the normal XIS flow.

Refresh, not data push

The event should usually say what is stale, not carry the full replacement model. This keeps the model source on the server and prevents a second client-side synchronization protocol from growing beside the normal page and frontlet lifecycle. 

import one.xis.Action;
import one.xis.ModelData;
import one.xis.Page;
import one.xis.RefreshEventPublisher;
import one.xis.RefreshOnUpdateEvents;

@Page("/cart.html")
@RefreshOnUpdateEvents("cart-updated")
class CartPage {
    private final RefreshEventPublisher refreshEvents;

    CartPage(RefreshEventPublisher refreshEvents) {
        this.refreshEvents = refreshEvents;
    }

    @ModelData("cart")
    CartView cart() {
        return cartService.currentCart();
    }

    @Action("addToCart")
    void addToCart(long productId) {
        cartService.add(productId);
        refreshEvents.publishToAll("cart-updated");
    }
}

Targets

Events can be broad or narrow. Some events are useful for all connected clients, some only for one authenticated user, and some only for one browser session. Choosing the narrowest useful target keeps the browser calm and avoids unnecessary refreshes. 

refreshEvents.publishToAll("cart-updated");
refreshEvents.publishToClient("cart-updated", clientId);
refreshEvents.publishToUser("cart-updated", userId);
refreshEvents.publishToAllUsers("account-notice-updated");

SSE transport

Server-sent events are a good fit for XIS refresh signals because they are one-way, HTTP-friendly, and simpler to operate than a bidirectional socket when the application only needs server-to-browser notifications. The event channel should stay small. Business data is still loaded by the normal XIS request that follows the event.

Consistency model

A refresh event is allowed to be slightly late. It should not make the browser invent state. If the browser misses an event, the next explicit navigation, action, or refresh request must still reconstruct the correct state from the server.
