Operations / Cloud native

Cloud native and native images

XIS can build JVM applications and GraalVM native executables. Native images are mainly about startup, memory, and deployment shape. Throughput-sensitive services should still be benchmarked in their target environment.

Native build tasks

The normal native task uses conservative defaults that work across machines. xisNativeCompileForHost is meant for deployments where the executable is built on the same CPU family on which it will run. It can pass host-oriented compiler options without making that the default for every build pipeline.

Optimization level should be configurable, but the default should remain aligned with GraalVM's documented recommendation. Users who care about a specific service must benchmark their actual workload. 

dependencies {
    implementation "one.xis:xis-boot-native"
}
./gradlew xisNativeCompile
./gradlew xisNativeCompileForHost
./gradlew xisNativeCompileForHost -PxisNativeImageArgs="-O3"

Native metadata

Native applications use generated catalogs and generated metadata instead of relying on runtime classpath scanning. This is a major part of the XIS native value: users should not have to hand-maintain reflection, proxy, resource, and component metadata. 

import one.xis.boot.XISBootApplication;

@XISBootApplication
public class Application {
}

The plugin uses the application class to generate the native entry point and the metadata catalogs. Application code does not call the generated native runner directly.

What native is good at

Native images are attractive for fast startup, lower memory use, small service processes, command-line helpers, and environments where many JVM processes would otherwise be too expensive. These are real operational benefits for XIS Boot applications and pure HTTP Controller services.

What native does not promise

A native executable is not automatically faster for long-running server throughput. The JVM can optimize hot code while the application is running, and that can beat compile-time optimization in some workloads. Choose native for startup time, memory footprint, packaging, and operational shape; benchmark long-running throughput in the target environment before treating native as the faster option.

Native database modules

Native images need database drivers and driver metadata at build time. On the JVM, a JDBC driver can often be a late runtimeOnly dependency and still be discovered through DriverManager or service loading. A native image is closed during the build: driver classes, service registrations, resources, reflection needs, and initialization behavior must already be reachable.

Add the normal persistence module plus the native driver module for the database used by the application: 

dependencies {
    implementation "one.xis:xis-boot-native"
    implementation "one.xis:xis-sql"
    implementation "one.xis:xis-boot-native-postgresql"
}

The available native SQL driver modules are xis-boot-native-h2, xis-boot-native-postgresql, and xis-boot-native-mariadb. For MongoDB applications use xis-boot-native-mongodb together with the MongoDB repository module. 

dependencies {
    implementation "one.xis:xis-boot-native"
    implementation "one.xis:xis-mongodb"
    implementation "one.xis:xis-boot-native-mongodb"
}

The complete guide has the full dependency matrix and background in Database Modules.

Reflection, records, and inner classes

Native-image support must include the classes used by controllers, DTOs, nested DTOs, records, validation objects, and response models. Inner classes matter because they often appear naturally in small page controllers and form models. The generated metadata must walk that structure instead of assuming one public top-level class per file.

HTTP services

The HTTP Controller API is an especially clean native-image target. A small service can depend on the HTTP runtime, validation, and its own business code without shipping the page framework JavaScript resources. If an application uses XIS pages, those resources are included because the browser runtime needs them. 

dependencies {
    implementation "one.xis:xis-http-controller-native"
}
