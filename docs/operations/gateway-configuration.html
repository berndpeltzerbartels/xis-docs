Operations / Gateway configuration

Gateway configuration

A XIS Boot application usually runs behind a gateway. Apache or nginx can terminate HTTPS, serve static pages, and forward the application paths to the local XIS process.

What must be proxied

XIS pages may be mounted below an application path such as /app-wizzard/, but the framework runtime also uses root-level resources and service endpoints. A gateway therefore has to forward the page path and the XIS runtime paths together.

For a page application behind /app-wizzard/, the important paths are:

Path Purpose
/app-wizzard/ The application page and its public resources.
/xis/ XIS internal endpoints, including server-sent events such as /xis/events.
/bundle.min.js The XIS browser runtime used by page applications.
/default-main.css Small framework CSS for XIS-owned browser elements.

Apache 2

Enable the proxy modules once and keep the application process bound to localhost. The example assumes the XIS application listens on 127.0.0.1:8081. 

sudo a2enmod proxy proxy_http
sudo systemctl reload apache2
ProxyPreserveHost On

ProxyPass        /app-wizzard/ http://127.0.0.1:8081/app-wizzard/
ProxyPassReverse /app-wizzard/ http://127.0.0.1:8081/app-wizzard/

ProxyPass        /xis/ http://127.0.0.1:8081/xis/
ProxyPassReverse /xis/ http://127.0.0.1:8081/xis/

ProxyPass        /bundle.min.js http://127.0.0.1:8081/bundle.min.js
ProxyPassReverse /bundle.min.js http://127.0.0.1:8081/bundle.min.js

ProxyPass        /default-main.css http://127.0.0.1:8081/default-main.css
ProxyPassReverse /default-main.css http://127.0.0.1:8081/default-main.css

Validate before reloading: 

sudo apache2ctl configtest
sudo systemctl reload apache2

nginx

nginx uses location blocks for the same split. SSE endpoints need buffering disabled so events reach the browser immediately. 

server {
    listen 443 ssl;
    server_name example.com;

    root /var/www/html/xis-docs;
    index index.html;

    location /app-wizzard/ {
        proxy_pass http://127.0.0.1:8081/app-wizzard/;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location /xis/ {
        proxy_pass http://127.0.0.1:8081/xis/;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_buffering off;
    }

    location = /bundle.min.js {
        proxy_pass http://127.0.0.1:8081/bundle.min.js;
        proxy_set_header Host $host;
    }

    location = /default-main.css {
        proxy_pass http://127.0.0.1:8081/default-main.css;
        proxy_set_header Host $host;
    }
}
sudo nginx -t
sudo systemctl reload nginx

Static documentation beside an app

Static documentation and a XIS application can share one host. Let the gateway serve static HTML directly and only forward the application and XIS runtime paths. That keeps documentation deploys independent from application deploys.

When Apache Server Side Includes are used for static documentation, keep SSI on the documentation directory and keep the XIS application behind proxy rules. The two concerns should not be mixed inside the application process.
