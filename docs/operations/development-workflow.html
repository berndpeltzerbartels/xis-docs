Development workflow

XIS is designed for a short backend-oriented development loop: write the controller, generate a starting template or test when useful, validate templates early, and keep executable tests next to the feature.

Start with a page controller

A XIS page starts with a controller annotated with @Page. The controller describes the URL, the model data, the form data, and the actions the browser can call. If these annotations are new to you, keep the Annotation reference, Pages and templates, and Forms and validation open while reading this example.

This small contact page intentionally contains a few things that the template generator can reflect later: a scalar model value, an array model value, a form object, validation annotations, and a form action. 

package example.contact;

import one.xis.Action;
import one.xis.FormData;
import one.xis.ModelData;
import one.xis.Page;
import one.xis.validation.EMail;
import one.xis.validation.Mandatory;

@Page("/contact.html")
class ContactPage {

    @ModelData("headline")
    String headline() {
        return "Contact us";
    }

    @ModelData("topics")
    String[] topics() {
        return new String[] { "Support", "Sales", "Partnership" };
    }

    @FormData("message")
    ContactMessage message() {
        return new ContactMessage("", "", "Support");
    }

    @Action
    void send(@FormData("message") ContactMessage message) {
    }

    record ContactMessage(
            @Mandatory String name,
            @Mandatory @EMail String email,
            @Mandatory String topic
    ) {
    }
}

@ModelData exposes values to the template. @FormData creates the server-side form object. @Action marks a method that a button or form can call. @Mandatory and @EMail are server-side validation annotations. The browser can show validation feedback, but the server remains authoritative.

Generate the template

Run xisTemplates after the controller shape exists. The task creates missing template files next to the controller. Existing templates are not overwritten. 

./gradlew xisTemplates

For the controller above, the task generated this actual file: 

<!DOCTYPE html>
<html lang="en" xmlns:xis="http://xis.one/schema">
<head>
    <meta charset="UTF-8"/>
    <title></title>
</head>
<body>


<section>
  <div>${headline}</div>
</section>

<section>
  <ul xis:foreach="topic:${topics}">
    <li>${topic}</li>
  </ul>
</section>

<form xis:binding="message">
  <xis:global-messages/>
  <div>
    <label for="message-name" xis:error-binding="name" xis:error-class="error">Name</label>
    <input id="message-name" type="text" xis:binding="name" xis:error-class="error"/>
    <div xis:message-for="name"></div>
  </div>
  <div>
    <label for="message-email" xis:error-binding="email" xis:error-class="error">Email</label>
    <input id="message-email" type="text" xis:binding="email" xis:error-class="error"/>
    <div xis:message-for="email"></div>
  </div>
  <div>
    <label for="message-topic" xis:error-binding="topic" xis:error-class="error">Topic</label>
    <input id="message-topic" type="text" xis:binding="topic" xis:error-class="error"/>
    <div xis:message-for="topic"></div>
  </div>
  <button type="button" xis:action="send">Save</button>
</form>

</body>
</html>

Notice how the generator mirrored the controller contract: headline became an expression, topics became a xis:foreach list, message became a form binding, the record components became inputs, and send became the form action. The result is not meant to be beautiful. It is meant to be executable wiring that you can now refine. For template rules, see Template syntax and Tags and attributes.

Generate the first test

The same controller can also get a starter integration test. In this probe project, running xisTemplates and xisTests as separate commands worked reliably: 

./gradlew xisTests

The task generated this actual test: 

package example.contact;

import one.xis.boot.test.XisBootTest;
import one.xis.context.IntegrationTestContext;
import one.xis.test.InTestContext;
import org.junit.jupiter.api.Test;

import static org.junit.jupiter.api.Assertions.assertNotNull;

/**
 * Integration test for ContactPage
 * @generated
 */
@XisBootTest
class ContactPageTest {

    private IntegrationTestContext context;

    @InTestContext
    private ContactPage contactPage;

    @Test
    void test() {
        // Edit url if you need path variables or query parameters
        var client = context.openPage("/contact.html");

        assertNotNull(client.getDocument());
        // TODO: Add more assertions
    }
}

@XisBootTest creates a XIS integration-test context. IntegrationTestContext opens a page the same way a browser would enter it. @InTestContext gives the test access to the managed page object. The generated assertion only proves that a document was rendered. From here, edit the test until it says what the page must really do: show the headline, render the topic list, submit invalid data, show validation messages, and call the action for valid data. See Testing from the start and Examples and tests.

Validate before running

After generation and after each meaningful template edit, run validation. It catches structural mistakes before you start the application: missing required framework attributes, invalid bindings, unknown xis:* elements, and model/form properties that do not match the controller contract. 

./gradlew xisValidate
./gradlew xisValidate --all-errors

Validation is not a replacement for tests. It proves that the template is structurally consistent with the server contract; tests prove that the feature behaves correctly.

Run locally

XIS Boot applications can be run through the Gradle plugin. Pass a port when the default port is already in use. 

./gradlew xisRun --port=9090

In XIS Boot development runs, public resources under src/main/resources/public are read from the source file when available. CSS and similar resource edits should therefore become visible after pressing the browser reload button. Packaged jars remain static, and Java/controller changes still require rebuilding or restarting the application.

The loop in practice

The practical rhythm is: 

./gradlew xisTemplates
./gradlew xisTests
./gradlew xisValidate
./gradlew test
./gradlew xisRun --port=9090

Then edit Java, templates, and resources together. Use browser reload for public resource changes, rerun xisValidate when template structure changes, and keep the test suite moving with the feature.
