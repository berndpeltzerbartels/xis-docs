Scheduled jobs

Scheduled jobs are a small but useful public feature. This chapter covers fixed delay, fixed rate, the scheduler component, and typical runtime implications.

Fixed delay and fixed rate

A fixed-delay job waits after one run has finished before scheduling the next. A fixed-rate job aims for a regular cadence. The difference matters when work can take longer than expected, so the code examples show both variants instead of treating them as interchangeable timers. 

import one.xis.context.Component;
import one.xis.context.Scheduled;

import java.util.concurrent.TimeUnit;

@Component
class ReminderJob {

    @Scheduled(initialDelay = 1, fixedDelay = 1, timeUnit = TimeUnit.MINUTES)
    void sendDueReminders() {
        reminderService.sendDueReminders();
    }
}
@Scheduled(initialDelay = 5, fixedRate = 10, timeUnit = TimeUnit.SECONDS)
void pollExternalStatus() {
    externalStatusService.poll();
}

Scheduler components

Scheduled work should live in ordinary components with explicit dependencies. That keeps jobs testable and avoids static utility code that is hard to replace in integration tests.

Interaction with UI refresh

A scheduled job can publish refresh events after it changes state. It should not talk directly to browser-specific structures. The browser should learn about the change through the same event and refresh model as any other server-side update.
