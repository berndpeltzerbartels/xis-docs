Operations / Gradle plugin

Gradle plugin and tooling

The Gradle plugin is part of the user-facing workflow: templates, tests, validation, runnable jars, native builds, local runs, catalogs, and smoke tests belong here.

Why the plugin exists

XIS intentionally moves some work to build time. Templates can be validated, catalogs can be generated, JavaScript resources can be bundled, and native-image metadata can be prepared before the application starts. The plugin keeps that workflow repeatable and visible. 

plugins {
    id "java"
    id "one.xis.plugin" version "0.18.0"
}

repositories {
    mavenCentral()
}

dependencies {
    implementation "one.xis:xis-boot"
}

Main tasks

TaskPurpose
xisTemplatesGenerate or prepare template-related build artifacts.
xisTestsGenerate starter tests and test support where the project uses that workflow.
xisValidateValidate templates and framework metadata early.
xisJarBuild a runnable XIS jar with the required runtime resources.
xisRunRun the application locally in the development workflow.
xisNativeCompileBuild a GraalVM native executable with portable defaults.
xisNativeCompileForHostBuild a native executable optimized for the current host.
 
./gradlew xisTemplates xisTests
./gradlew xisValidate
./gradlew xisJar
./gradlew xisRun

Validation belongs in the build

Template mistakes should fail before deployment. This includes missing bindings, invalid framework attributes, and template structures that a browser would parse differently from what the framework expects. XIS should be strict here because otherwise errors move from build time into user interactions.

Generated code and user code

Generated runners and catalogs are implementation details. Users should normally configure dependencies and tasks, not call generated native runners directly. The public API is the Gradle task and the application class.

Without the plugin

The framework should remain understandable without Gradle magic, but advanced behavior is less convenient without the plugin. Projects that bypass the plugin must take over the equivalent catalog, resource, validation, and native metadata work themselves.
