Core / Request lifecycle

Request lifecycle

XIS turns page loads, navigation, actions, forms, shared values, client state, and refresh events into one coherent browser/server flow.

Lifecycle picture

1. Browser eventLink, button, form, modal, drag/drop, or refresh signal.
2. Small XIS requestThe runtime sends the action, parameters, form data, and client id.
3. Controller methodJava code validates, calls services, and returns a response.
4. Server-rendered resultXIS renders the affected page, frontlet, modal, or model fragment.
5. DOM replacementThe browser runtime replaces the relevant elements; the shell stays alive.

Initial load

On the first request, XIS resolves the page, creates the model data, renders the HTML, and sends the page together with the runtime scripts needed for later interactions. A reload or deep link must be able to reconstruct the page without relying on hidden client state.

Action request

A click, form submit, drag-and-drop action, or modal command sends a small request to the server. The controller method runs, validation is applied where relevant, and the response describes the next UI state. The browser does not reimplement that decision. It applies the server response. 

import one.xis.Action;
import one.xis.ModelData;
import one.xis.Page;

@Page("/counter.html")
class CounterPage {
    private final CounterService counterService;

    CounterPage(CounterService counterService) {
        this.counterService = counterService;
    }

    @ModelData("count")
    int count() {
        return counterService.currentValue();
    }

    @Action("increase")
    void increase() {
        counterService.increase();
    }
}
<main>
  <strong>${count}</strong>
  <button xis:action="increase">Increase</button>
</main>

Partial updates

XIS can update a page region, a frontlet, a modal, or the whole page. The framework chooses the concrete DOM work from the response and the template structure. This is where XIS produces SPA behavior without asking the application developer to maintain a separate JavaScript state tree.

The phrase "whole page" still does not mean a browser full page reload. It means that the XIS runtime replaces the page content in the live browser document. The JavaScript runtime stays loaded, the application shell remains alive, and the transition is handled as a controlled DOM update.

Client state

Some browser state is inherently local: focus, input composition, scroll positions, and custom JavaScript widgets. XIS should preserve it when possible, but it should not become the source of business truth. Durable state belongs in the server model or in explicit browser storage fields.

Refresh events

Server-triggered refresh events invalidate page or frontlet data. They do not replace the lifecycle. After an event, the browser asks for the affected model through the usual XIS path and then applies the resulting update.
