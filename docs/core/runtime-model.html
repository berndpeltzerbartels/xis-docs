Core / Runtime model

Runtime model

XIS separates the programming model from the runtime integration. Most application code should depend on one runtime and then use the same controller annotations and templates.

Runtime choices

XIS Boot

The standalone runtime for lightweight XIS applications. Lightweight means a smaller infrastructure surface, not a smaller ambition. Classic web applications with templates, validation, authentication, SQL, and jobs are in scope.

XIS Spring

The right choice when the application is already a Spring Boot application or when the Spring ecosystem already provides important infrastructure the project relies on.

XIS Boot Native

The native-image path for standalone applications that should deploy as small native executables or compact containers.

HTTP Controller API

An advanced path for raw HTTP endpoints, special static apps, games, and native REST-like services. Normal XIS page applications usually do not need it.

Choosing between XIS Boot and Spring

If an application already lives in Spring Boot, staying with Spring is usually the right choice. The same is true when the application depends heavily on Spring ecosystem infrastructure that would otherwise have to be recreated as XIS components.

XIS Boot is the smaller standalone path. It is especially attractive for microfrontend-oriented applications and services where the web UI should stay close to the feature code, while the runtime should not bring a large general-purpose application platform with it. A small amount of ordinary infrastructure code is not a reason to abandon XIS Boot; the question is whether the project genuinely needs the breadth of the Spring ecosystem.

XIS Boot also has onboard support for the cloud-native path: the same standalone programming model can be built as a runnable JVM jar or, with XIS Boot Native, as a GraalVM native executable with generated metadata for components, resources, proxies, and reflection. That native path is not an add-on architecture; it is part of the reason to choose XIS Boot for deployable services with a deliberately small runtime and infrastructure surface.

If cloud-native deployment is a primary requirement and the application does not already need Spring, XIS Boot is usually the stronger recommendation. Spring can be used, but the native-image path tends to be heavier and more sensitive to the surrounding ecosystem. XIS Boot keeps the runtime surface smaller and the generated native metadata closer to the framework features the application actually uses.

Dependency shape

Normal applications choose one runtime dependency. That runtime brings the controller and validation APIs transitively. Lower-level modules are documented because they are public, but most applications should not assemble their own runtime from individual internal pieces. 

plugins {
    id "java"
    id "one.xis.plugin" version "0.19.0"
}

dependencies {
    implementation "one.xis:xis-boot"
    // or: implementation "one.xis:xis-spring"
    // or: implementation "one.xis:xis-boot-native"
}
import one.xis.boot.XISBootApplication;
import one.xis.boot.XISBootRunner;

@XISBootApplication
public class Application {
    public static void main(String[] args) {
        XISBootRunner.run(Application.class, args);
    }
}

Static resources

Public files live under src/main/resources/public and are served without the public path segment. XIS also uses public CSS and JavaScript resources when building the root page. Concrete public files can be served directly; the root path itself is not treated as a public directory.
