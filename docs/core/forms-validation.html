Forms and validation

XIS keeps validation rules on the server to avoid parallel client/server rule sets. The cost is an extra request when validation runs; the gain is one authoritative validation model.

Forms can cover normal input fields, nested objects, lists, type conversion, custom validators, validation messages, action validation failures, file uploads, and modal dialogs. The same binding model is used throughout.

Why server-side validation is the default

Client-side validation can feel immediate, but it often creates two rule systems: one in the browser and one on the server. Those systems drift. XIS favors one source of truth and then makes validation feedback part of the normal page update flow.

This does not prevent small browser-side conveniences. It only means correctness belongs to the server. The framework transports validation errors back to the UI and lets templates render them consistently.

A form is a Java contract plus HTML binding

The controller exposes a form object with @FormData. The template binds normal HTML controls to that object. Validation annotations stay on the Java type, so tests and runtime behavior use the same rules. 

import one.xis.Action;
import one.xis.FormData;
import one.xis.Page;
import one.xis.validation.EMail;
import one.xis.validation.MinLength;

@Page("/contact.html")
class ContactPage {

    @FormData("contact")
    ContactForm contact() {
        return new ContactForm("", "", "");
    }

    @Action("send")
    void send(@FormData("contact") ContactForm form) {
        // Called only after binding and validation have succeeded.
        contactService.send(form.name(), form.email(), form.message());
    }

    record ContactForm(
            @MinLength(1) String name,
            @EMail String email,
            @MinLength(10) String message) {
    }
}
<form xis:binding="contact">
  <label>Name<input xis:binding="name"></label>
  <label>Email<input xis:binding="email"></label>
  <label>Message<textarea xis:binding="message"></textarea></label>
  <button type="submit" xis:action="send">Send</button>
</form>

Modal forms are ordinary forms

Small dialog forms are common in business applications: edit one field, confirm a change, add a row, or collect a short decision without leaving the current page. XIS modals use the same controller, template, model data, form data, action, and validation concepts as normal pages.
